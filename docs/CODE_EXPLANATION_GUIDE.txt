1. PROJECT STRUCTURE
==================================================

HTML pages
- index.html: Home page.
- categories.html: Region chooser for organization pages.
- india.html: Indian organizations with search, filters, and modal details.
- international.html: International organizations with search, filters, and modal details.
- countries.html: Country-based organizations with search, filters, and modal details.
- case-study.html: Sustainability case study cards.
- heroes.html: Eco hero cards with images and share buttons.
- task.html: Dynamic task tracker page.
- feedback.html: Feedback form page.
- terms.html: Terms and Conditions page.
- privacy.html: Privacy Policy page.
- disclaimer.html: Disclaimer page.
- license.html: License page.
- offline.html: Offline fallback page.

CSS files
- css/defs.css: Global design tokens such as colors, spacing, fonts, shadows, radii, and breakpoints.
- css/base.css: Reset, base body styles, accessibility helpers, and container width rules.
- css/layout.css: Shared page layout such as header, nav, hero, footer, policy section, and general page blocks.
- css/components.css: Reusable UI pieces such as cards, buttons, modal, forms, filters, and PWA banners.
- css/pages.css: Page-specific rules such as counters, tracker UI, progress bars, and responsive adjustments.

JavaScript files
- js/navigation.js: Smooth scroll, active nav, mobile menu, header shadow, back-to-top button.
- js/modals.js: Card-driven modal and generic modal logic.
- js/search-filter.js: Search/filter logic for organization pages.
- js/forms.js: Feedback form validation and localStorage save.
- js/main.js: Main app bootstrap, PWA install/update logic, counters, scroll reveal, task tracker, social placeholders, service worker registration, performance hooks.
- js/service-worker.js: Actual caching logic for offline support.
- service-worker.js: Tiny root file that imports js/service-worker.js so the worker controls the whole site.

2. HTML ELEMENTS USED IN THIS PROJECT
==================================================

Document and metadata elements
- <!DOCTYPE html>: Tells the browser to use HTML5 mode.
- <html lang="en">: Root element. lang helps screen readers and search engines.
- <head>: Holds metadata, CSS links, manifest link, and deferred scripts.
- <meta>: Used for charset, viewport, SEO description, theme color, and PWA behavior.
- <title>: Browser tab title.
- <link>: Used for stylesheet loading, preload/prefetch hints, canonical URL, and manifest.
- <script>: Used for JSON-LD structured data and JavaScript files.

Layout and text elements
- <body>: All visible content.
- <header>: Top area of the page.
- <nav>: Navigation landmarks for accessibility.
- <main>: Main page content.
- <section>: Logical content groups.
- <div>: Generic container for layout.
- <span>: Small inline grouping element.
- <p>: Paragraph text.
- <h1>, <h2>, <h3>, <h4>: Headings used for hierarchy.
- <strong>: Strong emphasis inside text.
- <br>: Explicit line break, used in offline.html.

Interactive elements
- <a>: Navigation links and button-like links.
- <button>: Actions such as menu toggle, modal close, install app, progress popups, and form submit.
- <form>: Collects feedback.
- <label>: Gives accessible names to form fields.
- <input>: Search and form text/email fields.
- <textarea>: Multi-line feedback message.

List and media elements
- <img>: Images for heroes and case study cards.
- <ul> and <li>: License page bullet lists.
- <svg>, <rect>, <path>, <circle>: Inline vector logo in index.html.

Accessibility helpers
- aria-label: Gives extra meaning to buttons and links.
- aria-controls: Connects hamburger button to nav container.
- aria-expanded: Indicates mobile menu state intention.
- aria-current="page": Added by JS to the active nav link.
- aria-hidden: Used for modal visibility state.
- aria-modal: Marks modal as a true dialog.
- aria-labelledby: Connects dialog to its heading.
- aria-live: Makes tracker status text announce updates.
- aria-valuemin, aria-valuemax, aria-valuenow: Progressbar semantics.
- role="dialog", role="button", role="status", role="alert", role="progressbar", role="list", role="listitem": Extra semantic meaning for custom UI.

Custom data attributes
- data-category: Used by filters on organization pages.
- data-logo: Short text used to build a simple logo block in JS.
- data-focus: Comma-separated topics shown as tags inside modal.
- data-impact: Modal impact summary.
- data-website: External link shown in modal.
- data-verified-by: Present on one India card, but not currently used by JS.
- data-share-title and data-share-text: Present on hero share buttons, but there is no current JS listener wired to them.

5. SHARED ARCHITECTURE AND HOW FILES CONNECT
==================================================

Shared page shell
- Most pages use the same pattern: head -> skip link -> sticky header -> hero -> page content -> footer -> back-to-top button.
- The repeated shell is styled by defs.css, base.css, layout.css, and components.css.
- Shared JS is loaded with defer so HTML parses first and scripts run after the document is ready.

Organization listing flow
- india.html, international.html, and countries.html provide search input, filter buttons, org cards, and an empty modal shell.
- js/search-filter.js reads searchInput and filter buttons to hide/show cards.
- js/modals.js reads each org-card and uses its data-* attributes to build the modal content.

Task tracker flow
- task.html only contains an empty container with id="ecoTracker".
- js/main.js detects that container and builds the entire tracker UI dynamically.
- Progress is stored in localStorage under ecoTasks.

PWA flow
- manifest.json is linked in most pages.
- js/main.js handles install button text, install prompt, update prompt, and service worker registration.
- service-worker.js imports js/service-worker.js so the worker can be registered at the root scope.
- js/service-worker.js caches pages, CSS, JS, icons, and offline.html.

25) js/service-worker.js
Purpose
- Handles actual offline caching and fetch behavior.

Line walkthrough
- 4-5: Define cache names and versions.
- 8-40: ASSETS_TO_CACHE lists pages, CSS, JS, manifest, and icons that should be cached for offline use.
- 43-51: updateRuntimeCache() saves successful responses into the runtime cache.
- 54-68: install event
  - Opens the main cache.
  - Adds all static assets.
  - Calls self.skipWaiting() so the new worker activates quickly.
- 71-90: activate event
  - Deletes old cache versions.
  - Calls self.clients.claim() so the new worker starts controlling pages immediately.
- 93-144: fetch event
  - Ignores non-GET requests and cross-origin requests.
  - Returns cached responses first when available.
  - Refreshes cached items in the background.
  - Stores fresh network responses in the runtime cache.
  - Returns offline.html for failed document requests.
  - Returns a plain-text 503 response for other offline failures.
- 147-151: message event listens for SKIP_WAITING.

26) service-worker.js
Purpose
- Small root-level loader for the real service worker code.

Line walkthrough
- 1-2: Explain why this file exists.
- 3: importScripts('./js/service-worker.js') loads the real worker logic from the js folder.

Why it matters
- If the project registered only js/service-worker.js directly, the service worker scope would be limited to /js/.
- Putting this tiny file at the root allows the worker to control the whole site.

==================================================
8. IMPORTANT OBSERVATIONS ABOUT THE CURRENT CODE
==================================================

- The project is built as a static website with shared CSS and JS across many HTML pages.
- A lot of repeated header/footer markup exists across pages. This is normal in plain multi-page static sites, but it means edits often need to be repeated in several files.
- The organization pages are data-driven through HTML data-* attributes rather than through a backend.
- The feedback form and task tracker both store data only in localStorage. They do not submit to a server.
- The share buttons on heroes.html are present visually, but they are not currently connected to a click handler.
- Placeholder social links are intentionally blocked by JS and replaced with an alert.
- The homepage and task page are the most behavior-heavy pages because they rely on js/main.js.

9. SIMPLE READING ORDER FOR STUDYING THE PROJECT
==================================================

Recommended order for a beginner
1. Read index.html to understand the basic page skeleton.
2. Read css/defs.css and css/base.css to understand the design tokens and reset.
3. Read css/layout.css and css/components.css to understand how the UI is styled.
4. Read js/navigation.js because it is the simplest JS file.
5. Read js/forms.js and js/search-filter.js next.
6. Read js/modals.js to understand dynamic DOM building.
7. Read js/main.js last, because it is the main coordinator and contains the most logic.

End of guide.
